# Image Storage Architecture Summary (TanStack Start + S3-Compatible Storage + Proxy Route)

## 1. Goals & Constraints
- Must support **self-hosting** and **hosted** deployments.
- Must allow app to upload and retrieve **small–medium sized images**.
- Must store images in a **generic S3-compatible object store** (Garage, MinIO, R2, S3, SeaweedFS, etc.).
- Frontend is a **TanStack Start** project.
- Image URLs must be retrievable and renderable in the web app.
- Prefer a **proxy route** in TanStack Start (no public bucket, no direct S3 URLs).
- Image transforms can be added later, but not required now.

## 2. Storage Choice
Use **generic S3 compatibility** as the target API. This allows swapping the backend via environment variables without changing application code.

Supported backends:
- **Garage** (recommended for simple, lightweight self-hosting)
- **MinIO** (mature, more heavyweight)
- **AWS S3**, **Cloudflare R2**, **Backblaze B2 S3 API**, **SeaweedFS**, etc.

All backends are interchangeable as long as they expose the S3 API (PutObject / GetObject).

## 3. Application Architecture Overview

### Upload Flow
1. User uploads file → `/api/upload` server route in TanStack Start.
2. Route validates, processes, and **uploads the binary** to S3 with a generated key.
3. Route inserts a row into the `images` table:
   - `id` (UUID)
   - `storage_key` (S3 object key)
   - `mime_type`
   - Optional metadata (alt, width, height)
4. Response returns `{ id, storageKey }`.

### Serving Images via Proxy Route
Fetch image from S3 and stream through `/media/:id`.

## 4. Database Schema

```sql
CREATE TABLE images (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  created_at  timestamptz NOT NULL DEFAULT now(),
  owner_id    uuid,
  storage_key text NOT NULL,
  mime_type   text NOT NULL,
  width       int,
  height      int,
  alt         text,
  UNIQUE(storage_key)
);
```

## 5. S3 Client Configuration (src/server/s3.ts)

```ts
import { S3Client } from '@aws-sdk/client-s3'

export const s3 = new S3Client({
  region: process.env.S3_REGION,
  endpoint: process.env.S3_ENDPOINT,
  forcePathStyle: process.env.S3_FORCE_PATH_STYLE === 'true',
  credentials: {
    accessKeyId: process.env.S3_ACCESS_KEY_ID,
    secretAccessKey: process.env.S3_SECRET_ACCESS_KEY,
  },
})

export const S3_BUCKET = process.env.S3_BUCKET!
```

## 6. Upload Route (/api/upload)
- Accepts multipart form upload
- Validates mime types
- Uploads to S3 using `PutObjectCommand`
- Inserts DB row
- Returns `{ id, storageKey }`

## 7. Proxy Route (/media/:id)
- Looks up image in DB
- Fetches object via `GetObjectCommand`
- Streams bytes to browser with correct headers
- No CORS; storage backend hidden from frontend

Frontend uses:
```html
<img src="/media/{id}" />
```

## 8. Future Image Transforms
- Pre-processing on upload using Sharp
- On-the-fly transforms via query parameters
- External services (imgproxy, thumbor)

## 9. Why This Architecture Works
- Backend-agnostic & portable
- Secure, simple, scalable
- Allows transforms without schema changes
- Frontend stays clean: `/media/:id`
